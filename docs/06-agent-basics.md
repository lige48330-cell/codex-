# Agent 基础：Codex 不是普通聊天框

这页适合第一次接触 Codex、Agent、工具调用的小白。

目标很简单：

**先理解 Agent 是怎么工作的，再继续学习 Skill 和 MCP。**

Skill 和 MCP 是进阶内容，不是第一天必须学。

---

## 先记住一句话

普通聊天工具更像：

```text
你问一句，AI 回一句。
```

Codex 这样的 Agent 更像：

```text
你给一个目标，Codex 会理解任务、拆步骤、使用工具、观察结果，再决定下一步。
```

所以学习 Codex 的重点，不只是“怎么写提示词”，而是理解：

- Codex 会怎样理解目标。
- Codex 什么时候需要读文件、改文件、运行命令。
- Codex 为什么有时会先问你确认。
- 你该怎样给它边界和验收标准。

---

## 什么是 Agent

你可以先把 Agent 理解成一个“会干活的 AI 助手”。

| 组成 | 对小白来说是什么意思 |
|---|---|
| 模型 | 负责理解、推理、写内容、做判断 |
| 上下文 | 当前对话、文件内容、任务背景 |
| 工具 | 读文件、改文件、搜索、运行命令、打开浏览器等能力 |
| 计划 | 把大目标拆成一步步可执行的小任务 |
| 反馈 | 看工具结果是否成功，再调整下一步 |

一句话类比：

> 模型像大脑，工具像手脚，上下文像短期记忆，计划和反馈让 Codex 能循环推进任务。

具体一点说：如果你让 Codex 总结 `README.md`，它可能会先读这个文件，再整理重点，最后告诉你结果。这个过程比普通聊天多了“读取真实文件”和“根据结果继续判断”。

---

## Codex 的工作循环

Codex 做事时，通常不是“一次回答完”，而是一个循环：

1. **接收目标**：你告诉 Codex 想完成什么。
2. **理解背景**：Codex 读你的说明，也可能读项目里的 README、docs、配置文件。
3. **拆解步骤**：它判断先看什么、再改什么、最后怎样验证。
4. **选择工具**：例如读文件、搜索代码、运行测试、打开网页、调用 MCP 等。
5. **执行一步**：Codex 调用工具，拿到真实结果。
6. **观察结果**：如果命令失败，它会看报错；如果文件不存在，它会重新找路径。
7. **调整计划**：成功就继续，失败就换方法，风险高就问你。
8. **汇报结果**：最后告诉你做了什么、验证了什么、还有什么风险。

这就是 Agent 和普通 Chat 的核心区别。

## 小白最该学的是边界

第一天不用急着学 Skill、MCP、插件、Subagent。先学会把任务说清楚。

给 Codex 下任务时，尽量包含 5 件事：

| 要素 | 示例 |
|---|---|
| 目标 | “帮我把这份课程整理成小白能读懂的 Markdown。” |
| 背景 | “读者完全没写过代码，只知道 ChatGPT。” |
| 输入材料 | “只基于我给你的这几个本地文件，不要自己发挥。” |
| 输出格式 | “输出 4 个 Markdown 文件草稿，每个文件明确分隔。” |
| 约束 | “不要修改文件，只返回草稿；不确定先说。” |

一个好提示词可以这样写：

```text
请你作为 Codex，帮我整理一份小白教程。
读者完全不了解 Agent。
请先解释 Agent 和普通 Chat 的区别，再给出练习任务。
不要修改任何文件，只返回 Markdown 草稿。
如果发现材料互相矛盾，请标注“需要确认”。
```

---

## 你可以怎样指挥 Codex

### 让 Codex 先看，不要动

```text
请先阅读这个项目的 README 和 docs 目录，只总结结构，不要修改文件。
```

### 让 Codex 先给计划

```text
请先给我一个 3 步计划，每一步说明要验证什么。等我确认后再继续。
```

### 给 Codex 明确边界

```text
只能基于我提供的材料整理，不要联网搜索。
```

```text
可以修改 Markdown，但不要改代码文件。
```

```text
如果需要安装依赖、访问外部系统或使用 API Key，先停下来问我。
```

### 要求 Codex 给证据

```text
请说明你根据哪些文件得出这个结论，并列出对应文件名。
```

## Codex 可以做什么

在你允许的范围内，Codex 可以帮你：

- 阅读项目文件。
- 整理课程大纲。
- 起草 Markdown 文档。
- 修改代码或配置。
- 运行测试或命令。
- 分析报错。
- 用浏览器检查页面。
- 调用已配置的外部工具。

但要记住：

> Codex 能不能做某件事，取决于当前环境有没有对应工具、权限和上下文。

所以它不是“万能自动驾驶”，更像一个会执行任务的协作者。你负责给目标和边界，它负责推进和验证。

如果你准备让 Codex 修改文件，先确认回退方式：会 Git 就在任务前后做 checkpoint；不会 Git 就先复制一份文件夹备份。

## Claude Code、cc-haha 只作为类比

你可能在其他教程里见过 Claude Code 或 cc-haha。它们也常被拿来解释 Agent 工具，因为它们同样强调“目标 -> 工具 -> 反馈”的循环。

但本课程的主角是 **Codex**。

在这里，Claude Code 和 cc-haha 只作为背景类比，不作为默认推荐路径。本课程不推荐越权、绕过、破解、替换二进制，也不把非官方替代工具作为新手首选。

---

## 今天不要急着学 Skill 和 MCP

Skill 和 MCP 很有用，但它们属于进阶能力：

- **Skill**：把重复工作流程打包给 Codex 使用。
- **MCP**：让 Codex 连接外部工具、文档、浏览器、设计工具、系统等。

小白第一天只需要知道它们存在，不需要马上安装或配置。

更好的学习顺序是：

```text
先会给 Codex 下清楚任务
再会看 Codex 的计划和工具调用
再会要求验证结果
最后再学 Skill 和 MCP
```

## 本课练习

### 练习 1：让 Codex 解释自己要做什么

```text
请解释你作为 Agent 完成任务时，一般会经历哪些步骤。
请用小白能听懂的语言说明，不要超过 300 字。
```

观察它是不是提到了：理解目标、拆步骤、使用工具、检查结果。

### 练习 2：给 Codex 一个有边界的任务

```text
请帮我把“Agent 是会执行任务的 AI 助手”扩写成一段小白教程。
要求：
1. 不要提代码。
2. 不要超过 200 字。
3. 用生活类比。
4. 先给草稿，不要写入文件。
```

### 练习 3：让 Codex 先问问题

```text
我想写一份 Codex 入门课。
如果我的需求还不清楚，请先问我 3 个最关键的问题，不要直接开始写。
```

---

## 小结

今天只要记住三点：

1. **Codex 是 Agent，不只是聊天框。**
2. **Agent 的核心是循环：理解目标、调用工具、观察结果、继续调整。**
3. **小白先学会下清楚任务和设置边界，再学 Skill、MCP 这些进阶能力。**
